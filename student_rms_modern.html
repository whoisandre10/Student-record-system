Data Structure Justification
Why each structure was chosen for this system

Linked List

Primary storage for all student records. Each node holds one student and a pointer to the next, enabling dynamic memory growth without fixed-size array limits. Insertions are O(1) at head/tail; display traversal is O(n). Supports efficient removal without shifting elements.

Stack (LIFO)

Stores deleted records in last-in-first-out order to support Undo Delete. The most recently deleted student sits at the top, so undoing always restores the correct record. Push on delete, pop on undo. O(1) for both operations.

Queue (FIFO)

Manages the registration pipeline. New students are enqueued and wait for batch processing. This models real-world admission workflows where requests are handled in arrival order — first-come, first-served. Enqueue and dequeue are both O(1).

Hash Table

Provides O(1) average-case lookup by Student ID via a custom hash function with separate chaining for collision resolution. Critical for instant search when the college has thousands of records. Dramatically faster than linear search through the Linked List.

Binary Search Tree (BST)

Stores students ordered by ID for O(log n) search and sorted traversal (in-order = lexicographic ID order). Complements the Hash Table by supporting range queries and providing a secondary sorted index. Updates all three structures (LL, HT, BST) simultaneously on every insert, update, and delete to ensure consistency.
